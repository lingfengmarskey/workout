# CloudKit 同步测试设计书

发布前对 CloudKit 同步进行的多设备 / 中断 / 删除等验证。对应 issue #91（续自 #53）。
实现与策略见 [iCloud / CloudKit 同步设计](ICLOUD-SYNC-DESIGN.md)。

## 术语与准备

- **设备 A / 设备 B**：登录**同一个 iCloud 账号**的两台 iOS 设备（真机；照片/推送相关项建议真机而非模拟器）。
- **前台触发**：App 从后台回到前台会自动同步（`WorkoutApp` 的 `scenePhase == .active`）。
- **立即同步**：设置页手动触发。
- **状态位**：设置页展示 iCloud 账号状态、同步状态、最后成功时间、待上传记录数、待上传照片数、最近错误。
- 每个用例标注 **前置 / 步骤 / 预期**。除非特别说明，"同步"指等待一次自动或手动同步完成。

测试记录方式：在对应 sub-issue 里逐条勾选；发现问题就在该 sub-issue 下评论，附设备、步骤、实际结果。

---

## A. 开启与首次同步  （sub-issue: Group A）

- [ ] **A1 单设备首次上传**
  - 前置：设备 A 有若干计划、身体记录（含三角度照片）、饮食/锻炼记录，同步处于关闭。
  - 步骤：开启同步 → 走完确认页 → 等首次同步完成。
  - 预期：同步状态 `首次同步 → 正常`；待上传记录数、照片数归零；最后成功时间更新；无报错。
- [ ] **A2 第二台设备首次拉取**
  - 前置：A 已完成 A1；设备 B 同账号、未开启同步。
  - 步骤：B 开启同步 → 等首次同步完成。
  - 预期：B 合并出 A 的**全部**计划/记录/照片；**无重复记录**；三角度照片可显示；待上传数为 0。

## B. 日常增量同步  （sub-issue: Group B）

- [ ] **B1 单字段变更传播**：A 改一条身体记录（如体重）→ 同步；B 回前台后看到该变更。
- [ ] **B2 幂等性**：连续多次触发同步（前台切换/立即同步）→ 不产生重复记录、不重复上传未变化内容。
- [ ] **B3 手动同步与状态**："立即同步"可用并成功；最后成功时间、待上传数正确刷新。
- [ ] **B4 静默推送触发**（真机）：A 改记录并同步后，B 在后台收到变更、回前台已是最新（或收到推送后自动拉取）。

## C. 普通字段冲突（后写覆盖 / LWW）  （sub-issue: Group C）

- [ ] **C1 不同字段并发改**：A、B 均离线，各改同一条记录的不同字段 → 都联网同步 → 两处修改都在，无一方被旧值覆盖。
- [ ] **C2 同字段并发改**：A、B 离线改同一条记录同一字段 → 同步后保留 `updatedAt` 较新的值；较旧一方被正确覆盖为较新值（而非反向）。
- [ ] **C3 serverRecordChanged 重比较**：制造上传冲突（一方上传时另一方已改）→ 引擎重新比较、不盲目覆盖，最终收敛到较新值。

## D. 计划状态与"当前使用计划"  （sub-issue: Group D）

- [ ] **D1 多进行中计划不互相影响**：账号内存在多个进行中计划；同步**不因本机选择**自动暂停/完成/放弃其他计划。
- [ ] **D2 远端状态变更标注来源**：A 将某计划标记完成/暂停 → B 同步后状态更新，历史中标注"来自另一台设备"，**不播放** B 本机的仪式动画。
- [ ] **D3 当前使用计划为本地选择**：A、B 各自选择不同的进行中计划；两者互不同步、互不覆盖；今天/计划/进度页均使用各自本机选择。
- [ ] **D4 无选择时的提示**：合并出多个计划但本机未选当前计划时，提示用户从计划库选择，不按日期/顺序静默猜测。

## E. 照片同步与冲突  （sub-issue: Group E）

- [ ] **E1 三角度独立 + 去重**：正/侧/背分别上传下载、互不冲突；相同内容（SHA-256 相同）不重复上传。
- [ ] **E2 同角度较新者胜**：A、B 对同一记录同一角度各换一张照片 → 同步后保留 `updatedAt` 较新的照片。
- [ ] **E3 旧照片延迟清理**：新照片同步后，确认新 Asset 已下载且可解码，才清理本地旧文件（不出现"图裂"）。
- [ ] **E4 无关更新不重传**：只改体重/备注/另一角度，不触发本角度照片重新上传（观察待上传照片数不无故增加）。

## F. 删除与墓碑  （sub-issue: Group F）

- [ ] **F1 记录删除不复活**：A 删除一条记录 → 同步 → B 上该记录消失；且 A 的旧对象**不会被 B 重新上传复活**。
- [ ] **F2 历史计划删除**：删除某历史计划及其关联的身体/饮食/锻炼/照片 → 跨设备一致消失。
- [ ] **F3 照片删除传播**：删除某角度照片，通过版本化 `isDeleted` 传播；B 同步后该角度照片消失，非物理 Record 删除导致的不一致。
- [ ] **F4 同批墓碑防复活**：删除与旧记录在同一批同步到达时，墓碑优先，旧记录不被重新写入。

## G. 网络中断与健壮性  （sub-issue: Group G）

- [ ] **G1 断网失败可读 + 重试**：断网时触发同步 → 状态进入`需要处理`，展示可读错误 + 重试按钮；恢复网络后重试成功。
- [ ] **G2 中途中断安全续传**：同步进行中杀 App / 系统挂起 → 下次前台安全续传，不丢数据、不产生重复。
- [ ] **G3 大数据首次同步**：一台设备有大量记录 + 多张照片时开启同步 → 分批完成、UI 不卡死、最终一致，性能可接受。
- [ ] **G4 弱网**：限速/高丢包下多次同步仍能收敛，不出现死循环或重复上传。

## H. 账号与权限  （sub-issue: Group H）

- [ ] **H1 未登录/退出 iCloud**：设备未登录或退出 iCloud → 同步自动暂停，状态显示`未登录`；**不删除任何本地数据**。
- [ ] **H2 权限受限/暂时不可用**：iCloud 受限或暂时不可用 → 状态正确显示，自动暂停，不误删本地。
- [ ] **H3 账号切换**：切换到另一 Apple 账号 → 数据边界正确（不串号），状态正确。

## I. 关闭同步与删除云端数据  （sub-issue: Group I）

- [ ] **I1 停止此设备同步**：本机执行"停止此设备同步" → 保留本地数据和云端副本；设备 B 不受影响，仍能同步。
- [ ] **I2 删除云端数据**：执行"删除云端数据"（二次确认）→ 自定义 Zone 被删除；**本机数据默认保留**；文案说明其他设备再同步前可能仍有缓存。
- [ ] **I3 关闭 ≠ 删除云端**：确认"停止同步"和"删除云端数据"是两个独立动作，语义不混淆。

## J. 隐私与状态展示  （sub-issue: Group J）

- [ ] **J1 日志不含敏感数据**：抓取运行日志，确认不出现体重、腰围、备注、照片路径或 CloudKit payload。
- [ ] **J2 状态文案准确**：设置页各状态（账号/同步/最后成功时间/待上传数/最近错误）与真实一致；不宣称"实时同步"，正常文案为"会在网络和系统条件允许时自动同步"。
- [ ] **J3 仅 Private Database**：确认数据仅进入用户私有数据库（不使用 Public Database）。

## 验收总结（对应 #91 关闭条件）

- A–F 全部通过：多设备一致性、冲突、删除、照片核心正确。
- G 通过：中断/弱网/大数据下健壮。
- H–I 通过：账号边界、关闭与删除语义正确、不误删本地。
- J 通过：隐私与状态展示达标。
