# 实际进食记录与营养快捷录入 FSD

## 1. 文档信息

- 产品：减脂计划
- 功能：实际进食记录与营养快捷录入
- 状态：Draft，待产品确认后实施
- 平台：iOS
- 关联路线图：`ROADMAP.md` 的“P2：实际饮食与能量建议”

## 2. 功能目标

在保证营养计算可追溯、可修正的前提下，降低实际进食记录成本。

用户应该能够：

1. 快速记录经常吃的食物。
2. 扫描包装食品，获得包装上的营养数据。
3. 在识别失败或数据不完整时精确手动输入。
4. 在保存前查看并确认最终营养结果。
5. 将确认后的数据保存为最近使用、收藏或常用菜品模板。
6. 保留历史记录当时的营养快照，不受后续数据库变化影响。

## 3. 产品原则

### 3.1 精度优先，减少重复输入

有标准营养数据时优先使用标准数据；无法获得标准数据时允许估算；自动识别结果必须经过用户确认。

### 3.2 不让一种录入方式覆盖所有场景

包装食品、重复食物、家常菜和识别失败场景分别使用条码、模板、组合菜品和手动输入，不强制所有用户走同一条流程。

### 3.3 历史记录不可被静默改写

实际进食记录保存最终营养快照。模板、数据库或识别服务发生变化时，不能自动修改已经保存的历史记录。

### 3.4 自动识别是辅助，不是精确测量

OCR、条码数据库和照片估算都必须显示数据来源与可信度，不把估算结果描述为精确测量。

## 4. 功能范围

### 4.1 本 FSD 覆盖

- 最近使用食物。
- 收藏食物。
- 单个食物模板和组合菜品模板。
- 包装食品条码扫描入口。
- 包装营养成分表 OCR 识别入口。
- 统一识别结果确认和修正页面。
- 完整手动精确输入。
- 每份、每 100g、每 100ml、每包装之间的换算。
- 能量和三大营养素的统一计算。
- 数据来源、可信度和用户确认状态记录。

### 4.2 本阶段不覆盖

- 仅通过整盘食物照片精确计算热量。
- 自动识别烹饪油、酱汁、馅料和隐藏配料。
- 自动修改每日热量目标。
- 自动把等效活动建议加入锻炼计划。
- 面向公众的食品数据库运营后台。

## 5. 用户入口

在每日饮食详情页点击“添加实际进食”后显示：

```text
添加实际进食
├─ 最近使用
├─ 收藏食物
├─ 扫描包装条码
├─ 拍摄营养成分表
├─ 选择常用菜品
└─ 手动精确输入
```

默认优先显示最近使用和收藏内容，但不得隐藏手动输入入口。

## 6. 录入流程

### 6.1 最近使用和收藏

用户选择食物后，只输入实际数量或份数。系统自动带出食物名称、营养基准和营养值。用户可以修改营养值、取消收藏或删除模板。删除模板不得删除历史实际进食记录。

### 6.2 条码扫描

```text
扫描条码 → 查询食品数据 → 显示匹配商品和规格
→ 用户确认 → 输入吃了几份或多少克 → 统一确认页面
```

查不到商品、商品名称不匹配或规格不确定时，必须提供“拍摄营养成分表”和“手动输入”。条码数据不得未经确认直接写入历史记录。

### 6.3 营养成分表 OCR

```text
拍摄营养成分表 → 识别字段、数值、单位和营养基准
→ 显示原图与结果 → 用户确认或修正 → 输入实际摄入量
→ 统一确认页面
```

识别页面至少显示：原始照片、食物或品牌（如可获得）、能量、蛋白质、脂肪、碳水化合物、糖、膳食纤维、钠、营养基准和识别置信度。

识别失败时必须提供重新拍摄、手动输入和取消。

### 6.4 单个食物模板

用户第一次建立“熟米饭”等模板时确认营养值；以后只输入数量。模板至少包含名称、营养基准、能量、蛋白质、脂肪和碳水化合物。

### 6.5 组合菜品模板

组合模板可以包含多个食物：

```text
鸡胸肉便当
├─ 鸡胸肉 150g
├─ 熟米饭 150g
└─ 西兰花 100g
```

用户可以记录 0.5、1 或 1.5 份，修改或删除组成食物，并将修改后的组合另存为新模板。

### 6.6 手动精确输入

手动输入必须作为所有自动方式的保底入口，包含：食物名称、实际数量、单位、营养基准数量、每基准量能量、蛋白质、脂肪、碳水化合物，以及钠或食盐相当量。能量为必填，其余营养素在数据不可获得时允许为空。

## 7. 统一确认页面

所有来源最终进入同一确认页面：

```text
食物：熟米饭
实际数量：200g
营养基准：每100g

本次估算
能量：232 kcal
蛋白质：5.2g
脂肪：0.6g
碳水：51.8g

数据来源：用户模板
可信度：高

[保存] [返回修改]
```

用户可以修改数量、营养基准、营养值、名称和单位，也可以收藏、保存为模板或取消。保存后更新当前餐次、本餐合计和当日合计。

## 8. 计算规则

```text
倍数 = 实际数量 ÷ 营养基准数量

实际能量 = 每基准量能量 × 倍数
实际蛋白质 = 每基准量蛋白质 × 倍数
实际脂肪 = 每基准量脂肪 × 倍数
实际碳水化合物 = 每基准量碳水化合物 × 倍数
实际钠/食盐相当量 = 每基准量数值 × 倍数
```

整包装：`整包装能量 = 每份能量 × 包装内份数`。

部分包装：`实际能量 = 每 100g 能量 × 实际食用克数 ÷ 100`。

系统内部统一使用 kcal。识别到 kJ 时按 `1 kcal = 4.184 kJ` 转换，同时保存原始数值和单位。

## 9. 数据来源与可信度

### 9.1 来源

```text
manual          用户手动输入
template        用户确认的食物模板
barcodeDatabase 条码数据库
labelOCR        包装营养表 OCR
photoEstimate   食物照片估算
planned         从计划食物复制
```

### 9.2 可信度

```text
high    用户确认的包装标签、可靠数据库或已确认模板
medium  OCR 识别后未修改、用户自行建立的模板
low     照片估算或字段不完整的识别结果
```

未确认的 OCR 结果、低可信度且未修改的照片估算、缺少数量或营养基准的条目不得保存为正式历史记录。

## 10. 数据模型建议

### 10.1 FoodTemplate

```text
id, name, brand, barcode, locale
basisAmount, basisUnit
caloriesPerBasis, proteinPerBasis, fatPerBasis
carbohydratesPerBasis, sodiumPerBasis
source, confidence, isFavorite, lastUsedAt
createdAt, updatedAt
```

### 10.2 ActualFoodEntry 扩展

现有 `ActualFoodEntry` 保留营养计算字段，并建议增加：

```text
templateID, barcode, source, confidence, isConfirmed
originalUnit, originalNutritionJSON
```

历史条目保存最终快照，不直接引用可变的模板或数据库对象。

## 11. 校验与异常处理

以下情况阻止直接保存并提示用户：

- 食物名称为空。
- 实际数量或营养基准数量小于等于零。
- 能量为负数或无法解析。
- 每份和每包装的基准无法确认。
- 条码匹配商品与包装不一致。
- OCR 置信度过低。

以下情况允许保存但需要提示：蛋白质、脂肪或碳水缺失；只有部分字段被识别；识别结果经过用户修改。

## 12. 隐私和离线能力

- 包装照片默认保存在 App 私有目录。
- 营养表照片不得上传第三方服务，除非未来功能单独获得授权。
- 最近使用、收藏和手动模板必须离线可用。
- 无网络时不能阻止手动输入和历史记录保存。
- 删除模板不会删除已保存的实际进食记录。

## 13. 分阶段交付

### Phase 1：快捷模板

- 最近使用、收藏食物和单个食物模板。
- 统一确认页面。
- 数量换算和历史营养快照。

### Phase 2：包装食品识别

- 条码扫描入口和食品数据库适配层。
- 包装营养成分表 OCR。
- 中文、日文、英文营养字段映射。
- OCR 置信度和人工修正。

### Phase 3：组合菜品

- 常用菜品模板、份数缩放、组成食物编辑和模板版本管理。

### Phase 4：照片估算

- 食物照片候选识别、份量区间、可信度、用户确认和离线失败回退。

## 14. 验收标准

### 快捷记录

- 第二次记录同一食物时，最多三次点击即可进入数量输入。
- 修改模板不会改变历史记录。
- 收藏和最近使用离线可用。

### 包装识别

- 能识别常见中文、日文和英文营养字段。
- 能区分每 100g、每 100ml、每份和每包装。
- 识别结果确认后才能保存。
- OCR 失败时可以转入手动输入。

### 计算

- 每份、整包装和部分包装计算正确。
- g、ml、份不会被静默混用。
- kJ 与 kcal 转换正确。
- 本餐和当日累计实时更新。

### 安全

- App 重启后模板和历史记录不丢失。
- 删除模板不影响历史快照。
- 无网络时仍可完成手动记录。
