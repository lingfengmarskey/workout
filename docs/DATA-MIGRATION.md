# SwiftData 数据迁移策略

## 当前基线

`WorkoutSchemaV1`（1.0.0）是当前四个持久化模型的冻结基线：

- `WeightLossPlan`
- `DailyBodyRecord`
- `DailyMealPlan`
- `DailyWorkoutPlan`

App 通过 `WorkoutMigrationPlan` 创建 `ModelContainer`。数据库仍使用 SwiftData 的默认本地存储位置，不改变已有用户数据的位置。

## 模型升级流程

每次修改持久化字段、约束或关系时：

1. 不修改旧版本 Schema 内嵌套的模型定义或版本号。
2. 创建下一个 `VersionedSchema`，复制上一版模型后再修改，并为模型类型保留稳定的实体名称。
3. 按顺序将新 Schema 添加到 `WorkoutMigrationPlan.schemas`。
4. 只包含增加可选字段、删除字段等兼容变化时，添加轻量迁移阶段。
5. 需要改名、转换值或补齐必填字段时，添加自定义迁移阶段，并明确 `willMigrate` / `didMigrate` 的职责。
6. 用上一版本真实数据库副本启动新版本，验证计划、照片路径和每日记录均可读取。
7. 将供业务代码使用的顶层类型别名切换到最新 Schema 的模型类型。
8. 同时验证全新安装能够直接创建最新 Schema。

## 约束

- 已发布版本号不可复用或回退。
- 不通过删除数据库来处理迁移失败。
- 不随意修改模型类名；确需重命名时必须配置原始实体名称并测试迁移。
- 新增必填字段必须提供默认值，或通过自定义迁移为旧记录补值。
- 体型照片继续保存在文件系统中，迁移仅处理数据库内的文件引用。
- 合入模型变更前，PR 必须说明迁移类型以及旧数据验证结果。

## 测试清单

- 从上一版本数据库升级后记录数量一致。
- 当前计划及历史计划状态正确。
- 体重、腰围、饮食和锻炼历史可读取。
- 正面、侧面和背面照片路径仍有效。
- 升级后可以新增、编辑、删除并重新启动 App。
- 空数据库可以正常生成默认计划。
