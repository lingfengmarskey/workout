# 减脂计划 App MVP 产品需求文档

## 1. 产品概述

- 产品名称：减脂计划（暂定）
- 首期平台：iOS
- 首期用户：开发者本人
- 产品阶段：MVP
- 默认计划周期：8 周
- 数据单位：kg、cm、kcal、g、分钟、步数

本产品用于把减脂计划、每日执行记录和每周复盘集中到一个 App 中。核心不是建立复杂营养数据库，而是降低每天执行和记录的成本。

## 2. 产品目标

用户每天打开 App 后，应能快速知道：

1. 今天应该吃什么、吃多少。
2. 今天应该完成什么锻炼和多少步数。
3. 今天是否已经记录体重和体型。
4. 当前体重是否符合计划趋势。
5. 本周饮食和训练执行得怎么样。

### 成功标准

- 每日记录可在 2 分钟内完成。
- 首页直接展示当天全部任务。
- 支持至少连续 8 周记录。
- 能查看计划体重、实际体重和 7 日平均趋势。
- 核心功能无网络可用。
- 体型照片默认仅保存在本地或个人 iCloud。

## 3. 产品原则

1. **记录最少必要信息**：不要求详细记录每种调味料或每次心率变化。
2. **计划与执行分离**：系统提前生成计划，用户主要负责确认完成情况。
3. **关注长期趋势**：重点展示 7 日平均体重和每周执行率。
4. **隐私优先**：体重、腰围和照片不得默认上传第三方服务器。
5. **个人版优先**：MVP 不做社交、排行榜、订阅和公开分享。

## 4. 首期用户与默认目标

- 男性，35 岁
- 身高 180 cm
- 起始体重 97 kg
- 8 周阶段目标 88～90 kg
- 长期目标 80 kg
- 主要方式：饮食控制、步行、有氧和全身力量训练

## 5. 核心使用流程

### 5.1 首次使用

1. 创建减脂计划。
2. 输入身高、起始体重、目标体重、开始日期和计划周期。
3. 设置每日热量、蛋白质和饮水目标。
4. 选择默认 8 周模板或手动创建。
5. 系统生成每日体重目标、饮食计划和训练计划。

### 5.2 每日使用

1. 早晨记录体重。
2. 按需记录腰围、睡眠和体型照片。
3. 查看早餐、午餐、晚餐和加餐计划。
4. 每顿饭后标记完成状态。
5. 查看并完成当天训练与步数。
6. 晚间记录实际训练时长、步数、饥饿感和身体状态。

### 5.3 每周使用

1. 查看周平均体重和本周下降幅度。
2. 查看饮食与训练执行率。
3. 查看腰围、睡眠、饥饿感和精神状态。
4. 根据规则决定是否调整下一周计划。

## 6. 信息架构

底部四个 Tab：

1. **今天**：今日体重、饮食、训练与完成度。
2. **计划**：日历、每日饮食计划和每日训练计划。
3. **进度**：体重趋势、腰围趋势、体型照片和每周复盘。
4. **设置**：当前计划、提醒、单位、导出和隐私。

- 替换单个食物

#### 7.3.1 实际进食明细与能量计算（升级功能）

用户在早餐、午餐、晚餐或加餐中，不仅可以标记完成状态，还可以记录本次实际吃掉的食物。

录入方式：

- 手动搜索或输入食物名称。
- 输入重量、份数及单位，例如熟米饭 200 g、鸡肉 100 g。
- 从计划食物复制到实际记录后修改数量。
- 从最近吃过和收藏食物中快速选择。
- 同一餐可以持续增加、修改或删除多个食物条目。

每个实际食物条目至少包含：

- 食物名称。
- 实际数量和单位。
- 每 100 g 或每份能量。
- 本条目估算能量 kcal。
- 蛋白质、碳水和脂肪；数据不可得时允许为空。
- 数据来源：内置、用户填写、照片估算或第三方数据库。
- 估算可信度和用户是否已确认。

系统实时计算并展示：

- 本次进食总能量。
- 蛋白质、碳水和脂肪合计。
- 与本餐计划值的差异。
- 当日累计摄入和当日目标剩余量。

营养值均标注为估算结果。用户修改食物或数量后，合计应立即更新；保存历史记录后，后续营养数据库更新不得静默改变原记录。

#### 7.3.2 食物拍照估算（升级功能）

用户可以拍摄本次进食的照片，由识别能力生成候选食物、份量和营养估算。

流程：

1. 拍照或从相册选择。
2. 识别照片中的一个或多个候选食物。
3. 展示食物名称、估算重量、能量区间和可信度。
4. 用户逐项确认、修改、补充或删除。
5. 用户确认后才写入实际饮食记录并计算本餐合计。

约束：

- 不把单张照片结果描述为精确测量。
- 无法可靠识别烹调油、酱汁、馅料或遮挡食物时必须提示。
- 低可信度结果不得自动保存。
- 原始食物照片默认保存在 App 私有目录；调用云端识别前必须单独告知并获得同意。
- 离线或识别失败时仍可切换到手动输入。

#### 7.3.3 等效运动建议与加入今日计划（升级功能）

保存实际饮食后，App 可以把本餐能量换算为若干可选的运动示例，例如快走、慢跑、骑车、游泳或椭圆机，并展示估算时长或距离。

建议计算输入：

- 本餐估算总能量。
- 用户当前体重。
- 运动类型和强度。
- 对应活动的 MET 或经过验证的能量消耗模型。

展示规则：

- 同时提供至少 3 种不同冲击程度的运动选项。
- 结果使用区间，例如“快走约 55～70 分钟”，而不是伪精确数值。
- 明确说明实际消耗会受体重、速度、坡度、技术和设备影响。
- 文案使用“等效活动参考”，不得暗示用户必须通过运动抵消进食，也不得制造羞耻感。
- 建议不得超过用户当前运动能力或已有安全限制；存在疼痛或异常症状时不生成高强度建议。

用户操作：

- 忽略建议，不修改运动计划。
- 选择一种运动并调整时长、距离或强度。
- 确认后把它作为“饮食后追加活动”加入今日运动计划。
- 追加内容不得覆盖原训练，必须标明来源餐次、估算消耗和创建时间。
- 用户可以单独完成、编辑或删除追加活动。
- 删除或修改原饮食记录时，提示用户是否同步更新关联活动，不自动静默修改。

首版只提供参考换算，不根据运动建议自动增加当天可摄入热量，也不把“摄入减运动”作为唯一健康指标。

- 腰围变化
- 饮食执行率
- 训练执行率

- 平均饥饿感
- 平均晨起精神

调整规则：

- 每周下降约 0.4～1.2 kg 且状态正常：维持计划。
- 连续两周下降不足 0.4 kg：建议每日减少约 150 kcal，或增加约 2000 步。
- 每周下降超过 1.2 kg 且有疲劳、头晕、饥饿或训练退步：建议增加 150～200 kcal 或降低训练量。
- 系统只给建议，修改计划必须由用户确认。

用户可开关、修改时间、区分工作日与周末并推迟提醒。

### 7.8 计划管理

计划字段：

- 名称
- 开始和结束日期
- 起始体重
- 阶段目标体重
- 长期目标体重
- 每日热量
- 每日蛋白质
- 每日饮水
- 状态

计划状态：未开始、进行中、已暂停、已完成、已放弃。

计划状态属于计划本身，并会随计划同步到同一 iCloud 账号。多设备场景下允许同时存在多个“进行中”计划；“进行中”不再代表某台设备当前正在查看的计划。

每台设备额外维护一个本地的“当前使用计划”选择。该选择不上传到 iCloud，因此 A、B 两台设备可以分别使用不同计划。今天、计划、进度和本地提醒只读取当前使用计划，不再按状态自动选择计划。

#### 状态语义

- 未开始：计划已经创建，但用户尚未正式开始执行。
- 进行中：用户主动开始且尚未暂停、完成或放弃的计划；同一账号和同一设备上可以同时存在多个进行中计划。
- 已暂停：用户主动暂时停止，保留以后恢复的可能。
- 已完成：用户主动确认计划结束；不得因为创建新计划而自动完成。
- 已放弃：用户明确决定不再继续该计划；不得由系统自动设置。

#### 状态转换原则

- 每一次状态改变都必须由用户主动发起并确认，系统不得根据日期、创建新计划或其他事件自动改变状态。
- 创建新计划时不自动改变任何已有计划的状态；新计划创建后默认设为本机当前使用计划，用户随后可以在计划库切换。
- 恢复暂停计划只改变被恢复计划的状态，不需要先处理其他进行中计划；恢复后默认设为本机当前使用计划，用户随后可以在计划库切换。
- 切换当前使用计划只改变本机选择，不改变任何计划状态，不播放完成、暂停或放弃动画。
- 状态改变只影响计划生命周期，不得删除该计划的体重、围度、照片、饮食、锻炼或周报。
- 状态转换完成后应立即保存，并在历史计划中保持可追溯。

#### 计划库与当前使用计划

- 计划库展示本机已有的全部计划，以及从 iCloud 合并下来的全部计划。
- 首次开启同步时，云端计划和本机计划都保留；系统不得因为发现云端计划而删除或覆盖本机默认计划。
- 计划库明确标记每个计划的生命周期状态和“当前使用”状态；账号级同步状态在 iCloud 同步页面展示。
- 每台设备只能有一个当前使用计划，但可以存在多个进行中的计划。
- “设为当前使用计划”只保存设备本地的计划 ID，不改变计划的进行中、暂停、完成或放弃状态，也不上传到 iCloud。
- 今天、计划、进度和本地提醒都使用当前使用计划；没有选择时显示计划选择提示，不按日期或数组顺序静默猜测。
- 选择暂停、完成或放弃的计划作为当前使用计划时，必须先完成原有的恢复流程；恢复计划仍需用户确认并显示反馈动画。

#### 仪式感与反馈动画

- 完成计划：二次确认后显示全屏庆祝动画，突出计划名称、执行天数、累计减重等成果，并给予明确、真诚的鼓励。动画结束前不得阻塞数据保存。
- 暂停计划：确认后使用平静、克制的过渡动画，说明记录已安全保留，可以随时恢复。
- 恢复计划：确认后将计划设为进行中，并使用重新出发的正向动画；不要求处理其他进行中的计划。
- 放弃计划：先展示明确确认页，解释所有历史记录仍会保留；确认按钮使用“确认放弃计划”等完整文案，避免误触。完成后可使用低落但温和的动画，不得羞辱、责备或通过负罪感强迫用户继续。
- 动画应支持“减少动态效果”辅助功能；开启后以淡入、图标和文字反馈替代大幅移动、粒子或缩放。
- 所有结果动画均应提供明显的关闭按钮，不得强制观看完整动画。

#### 状态转换验收标准

1. 不存在任何无用户确认的计划状态改变。
2. 放弃和完成计划至少需要一次清晰确认；切换当前使用计划不改变生命周期状态，但应提供明确的选择反馈。
3. 完成计划后展示庆祝反馈；放弃计划后展示温和的告别反馈。
4. 用户取消确认时，计划状态和数据均保持不变。
5. 状态改变后，旧计划及其全部历史数据仍可访问。
6. VoiceOver 能读出操作目的、风险和结果；减少动态效果开启时仍有完整反馈。

#### 历史计划删除

- 只有已暂停、已完成或已放弃的历史计划可以删除；进行中计划必须先由用户手动改变状态。
- 删除入口放在历史计划详情底部，使用红色破坏性按钮，不放在列表滑动手势中，避免误触。
- 点击删除后必须显示居中 Alert，明确列出将永久删除计划、身体记录、体型照片、饮食记录、锻炼记录和周报数据。
- 用户必须点击完整文案“永久删除计划”才能继续；取消后不得改变任何数据。
- 确认后删除该计划及所有通过 `planID` 关联的数据，并清理 App 私有目录中的正面、侧面和背面照片文件。
- 删除不影响其他历史计划或当前进行中的计划。
- 数据库删除失败时必须回滚并提示错误，不得从页面消失或显示删除成功。
- 删除成功后返回历史计划列表；删除操作不可撤销。

#### 历史计划删除验收标准

1. 所有进行中的计划都没有删除入口；必须先由用户手动改变该计划状态。
2. 未确认 Alert 时不能删除任何数据。
3. 删除后，计划及其身体、饮食和锻炼数据无法再查询到。
4. 关联的本地体型照片文件被清理。
5. 其他计划的数据和照片不受影响。

## 9. 计算规则

- BMI：当前体重除以身高米数的平方。
- 计划体重：MVP 使用起始体重到阶段目标之间的线性下降曲线。
- 7 日平均：使用当日及之前最多 7 条有效记录，缺失日不按 0 计算。
- 累计减重：起始体重减当前实际体重。
- 饮食执行率：完成 1 分，部分完成 0.5 分，其他 0 分。
- 训练执行率：完成 1 分，部分完成 0.5 分，未完成 0 分；计划休息日单独处理。

## 10. 隐私与安全

- 默认不注册账号。
- 数据优先保存在本地。
- 体型照片不得上传第三方分析服务。
- 开启 iCloud 前明确提示同步范围，体型照片必须与对应记录一起同步。
- iCloud 同步默认关闭，本地 SwiftData 始终作为离线可用的主存储。
- 停止本机同步与删除 iCloud 数据必须是两个独立操作，后者需要二次确认。
- 同步失败不得删除、移动或阻止访问本地数据，并应显示最后成功时间与可重试错误。
- App 进入非活跃状态时，以不透明隐私遮罩隐藏体型与健康内容。
- 支持 Face ID、Touch ID 或设备密码隐私锁；默认关闭，由用户主动开启。
- 开启或关闭隐私锁前均须完成一次系统身份验证。
- 隐私锁开启后，冷启动及从后台返回时必须重新验证；失败或取消时不得显示记录内容，并提供重试入口。
- App 不读取或保存任何生物识别数据，仅接收 iOS 返回的验证结果。
- 删除计划和照片需要确认。

## 14. MVP 验收标准

1. 可以创建一个 8 周计划并生成 56 天数据。
2. 可以每日记录体重、腰围、睡眠和照片。
3. 可以查看每天四个餐次并标记状态。
4. 可以查看每日训练、目标步数并记录完成情况。
5. 可以显示计划、实际和 7 日平均体重趋势。
6. 可以自动汇总每周体重变化及饮食、训练执行率。
7. 重启 App 后数据不丢失。
8. 无照片权限或无网络时 App 不崩溃。
