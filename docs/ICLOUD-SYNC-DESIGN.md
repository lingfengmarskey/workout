# iCloud / CloudKit 同步设计

## 1. 目标

在不改变离线优先体验的前提下，让同一 iCloud 账号下的多台 iPhone 同步：

- 减脂计划及状态
- 每日身体、饮食和锻炼记录
- 正面、侧面和背面体型照片
- 记录删除与历史计划删除

同步默认关闭。用户必须在设置中主动开启，并在上传任何数据前看到同步范围、隐私边界和关闭方式。

## 2. 不采用 SwiftData 自动 CloudKit 镜像

当前模型包含 `@Attribute(.unique)`。SwiftData 的自动 CloudKit 同步不支持唯一约束；自动镜像也难以提供可靠的单次同步完成回调、逐项错误、用户可控的运行期开关，以及照片文件与数据库记录的一致迁移。

本项目采用：

```text
SwiftData 本地数据库（始终是主存储）
            ↕
      CloudSyncEngine
            ↕
CloudKit Private Database + CKAsset
```

这样不会在开启或关闭同步时替换 `ModelContainer`，也不会把用户已有的本地数据迁入另一个 SQLite 文件。

## 3. 数据边界

### 3.1 同步的数据

- `WeightLossPlan`
- `DailyBodyRecord`
- `DailyMealPlan`
- `DailyWorkoutPlan`
- 体型照片原始压缩文件
- 删除墓碑和同步版本信息

### 3.2 暂不同步的数据

- Face ID / 隐私锁开关
- 通知时间和通知开关
- 页面显示方式、图表时间粒度等 UI 偏好
- HealthKit 授权状态
- 临时缓存、缩略图和照片分析结果

这些配置属于单台设备，避免一台设备的设置意外改变另一台设备。

## 4. CloudKit 结构

使用用户私有数据库和自定义 Record Zone：

- Container：`iCloud.com.lingfengmarskey.workout`
- Zone：`WorkoutPrivateZone`
- Record Name：`<类型前缀>-<UUID>`，在所有设备上稳定一致

Record 类型：

| Record 类型 | 主要字段 |
|---|---|
| `WLPlan` | 计划全部持久化字段、`updatedAt`、`schemaVersion` |
| `WLBody` | 身体记录字段、三个照片版本、`updatedAt` |
| `WLMeal` | 饮食计划及执行字段、`updatedAt` |
| `WLWorkout` | 锻炼计划及执行字段、`updatedAt` |
| `WLPhoto` | 所属身体记录、角度、`CKAsset`、哈希、更新时间 |
| `WLTombstone` | 原 Record Name、类型、删除时间、设备 ID |

照片单独作为 `WLPhoto`，避免修改文字或体重时重复上传大文件。每张照片计算 SHA-256；哈希相同则不重复上传。

## 5. 本地模型升级

建立 `WorkoutSchemaV2`，通过轻量迁移增加：

- 所有可同步实体的 `updatedAt`
- `syncRevision` 或等价的本地变更标记
- `SyncTombstone`：记录已删除对象，直到服务器确认
- `CloudSyncState`：保存 Record Zone change token、最后成功时间和错误摘要
- 照片内容哈希及云端版本标记

不删除现有 UUID，也不在这一步移除唯一约束。已有数据迁移后标记为“等待首次上传”。

## 6. 开启同步流程

1. 用户进入“设置 → iCloud 同步”。
2. App 检查 iCloud 账号状态；不可用时说明原因，不改变设置。
3. 展示确认页，明确包含体型照片、仅进入用户私有数据库、可能消耗流量。
4. 用户确认后创建自定义 Record Zone 和订阅。
5. 先拉取云端变更，再合并本地数据，避免覆盖另一台设备已有内容。
6. 上传本地待同步记录与照片。
7. 首次拉取和上传均完成后，才把状态显示为“同步已开启”。
8. 中途失败时保留本地数据，状态显示“尚未完成”，允许重试。

不得因为没有网络、iCloud 空间不足或 CloudKit 错误而阻止本地记录。

## 7. 增量同步

触发时机：

- App 启动及进入前台
- 本地保存后进行防抖上传
- CloudKit 静默推送到达
- 用户下拉或点击“立即同步”
- 网络恢复后重试

使用 Record Zone change token 增量拉取。上传按小批次执行；照片与结构化记录分别重试。同步引擎必须可重复执行，同一批任务重放不得制造重复记录。

## 8. 冲突策略

### 普通字段

- 使用对象级 `updatedAt` 比较，较新的版本胜出。
- 时间完全相同时，使用稳定的设备 ID 作为最终排序，确保各设备得到一致结果。
- 合并前保存服务器版本；上传时遇到 `serverRecordChanged` 必须重新比较，不可盲目覆盖。

### 计划状态

计划的完成、暂停、恢复、放弃仍必须由用户操作触发。远端状态变化可以同步到本机，但 App 要在历史记录中标明变更来自另一台设备，不播放本机仪式动画。

### 照片

- 不同角度互不冲突。
- 同一记录、同一角度发生两次修改时，保留更新时间较新的照片。
- 旧照片延迟删除，确认新 Asset 已下载且可解码后再清理本地文件。

### 删除

删除必须同步墓碑，不能把另一台设备的旧对象重新上传。墓碑至少保留 90 天；超过期限后仅在所有已知变更都已同步的情况下清理。

## 9. 关闭与删除云端数据

关闭页面提供两个明确动作：

1. **停止此设备同步**：保留本地数据和云端副本；其他已开启设备不受影响。
2. **删除 iCloud 中的减脂计划数据**：二次确认后删除自定义 Zone；本机数据默认保留，并说明其他设备在再次同步前可能仍有缓存。

不得把关闭开关等同于删除云端数据。

退出 iCloud、账号切换或权限受限时自动暂停同步，不删除任何本地内容。

## 10. 设置页与状态

设置页展示：

- iCloud 账号：可用 / 未登录 / 受限 / 暂时不可用
- 同步：关闭 / 首次同步 / 正常 / 暂停 / 需要处理
- 最后成功同步时间
- 待上传记录数和照片数
- “立即同步”“停止此设备同步”“删除云端数据”
- 最近一次可读错误和重试按钮

App 不宣称“实时同步”。正常文案使用“会在网络和系统条件允许时自动同步”。

## 11. 安全与隐私

- 仅使用 CloudKit Private Database，不使用 Public Database。
- 照片本地继续使用完整文件保护。
- 临时 CKAsset 文件用完整文件保护，并在上传后清理。
- 日志不得记录体重、腰围、备注、照片路径或 CloudKit payload。
- Face ID 锁仅保护本机 App 界面，不应被描述为 CloudKit 加密功能。
- 开启前明确告知：数据会存入用户的 iCloud，并受其 Apple 账号和设备安全设置保护。

## 13. 验收标准

- 未开启同步时，行为与当前版本完全一致。
- 开启失败不会移动、删除或阻止访问本地数据。
- 两台设备可以同步计划、记录、状态和三类照片。
- 离线修改后恢复网络不会产生重复对象。
- 同一对象发生冲突时结果确定，且不会静默丢失较新修改。
- 删除记录或计划后，不会被另一台离线设备复活。
- 照片记录与照片文件最终一致；下载失败时不显示为已完成。
- 用户能区分“停止同步”和“删除 iCloud 数据”。
- App 可展示最后成功同步时间和可操作的错误状态。
- CloudKit schema 提升到 Production 前完成迁移与双设备真机测试。

## 14. 暂不纳入首版同步

- 多人共享计划
- Web 或 Android 客户端
- 冲突版本浏览与人工逐字段合并
- 自建服务器
- 跨 Apple ID 数据转移
